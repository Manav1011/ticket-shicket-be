# Fix: Claim Link Isolation + Security Issues

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix 5 bugs reported on 2026-04-26:
1. `claimLink` exposed in organizer→customer response (public URL)
2. `holderId`/`eventDayId` exposed in claim redemption response (security)
3. `holderId`/`eventDayId` also exposed in reseller response (same issue)
4. Self-transfer blocked in `split_claim` (should be allowed)
5. Ticket count aggregates ALL tickets (each claim link should be isolated)

**Architecture:**
- Add nullable `claim_link_id` to `tickets` table — tags tickets with the claim link that created them
- New transfers: tickets are tagged with `claim_link_id` at ownership transfer time
- On redemption: query only tickets matching `claim_link_id` (primary) OR legacy fallback (`claim_link_id IS NULL AND owner_holder_id match`)
- Legacy: tickets without `claim_link_id` fall back to old query behavior — existing claim links continue to work
- Organizer and reseller transfers: use `update_ticket_ownership_batch` (not raw SQL) so `claim_link_id` is consistently set

async def update_ticket_ownership_batch(
    self,
    ticket_ids: list[UUID],
    new_owner_holder_id: UUID,
    claim_link_id: UUID | None = None,
) -> None:
    if not ticket_ids:
        return

    values = {
        "owner_holder_id": new_owner_holder_id,
        "lock_reference_type": None,
        "lock_reference_id": None,

    )

## Legacy Data Handling

After code deployment, existing tickets from claim link transfers will have `claim_link_id = NULL`. The fallback query in `get_claim_redemption` ensures these continue to work:
- `claim_link_id IS NULL AND owner_holder_id = to_holder_id`

This means legacy claim links aggregate ALL untagged tickets — which is the current (buggy) behavior. But since we're adding `claim_link_id` to newly transferred tickets, each new transfer will be properly isolated. Over time, as old tickets are re-transferred, they'll get properly tagged.

**Long-term backfill** (separate migration, not part of this plan):
```sql
-- For each active claim link, tag untagged tickets owned by to_holder_id
UPDATE tickets t
SET claim_link_id = cl.id
FROM claim_links cl
WHERE t.claim_link_id IS NULL
  AND t.owner_holder_id = cl.to_holder_id
  AND t.event_day_id = cl.event_day_id
  AND t.status = 'active'
  AND cl.status = 'active'
```
